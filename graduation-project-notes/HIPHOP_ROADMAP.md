# Jasmine HipHop — 嘻哈垂直推荐 × 可视化实施路线图

> 面向：毕业设计答辩 + 长期产品演进  
> 日期：2026-06-07  
> 版本：v2.0  
> 定位：推荐为主，可视化为辅；数据可视化 + 播放器动效双线并行

---

## 一、为什么要加可视化

**对答辩**：可视化是老师一眼能看懂的东西。算法对不对他不知道，但图好不好看他立刻知道。

**对论文**：「数据可视化」是一整个章节的天然主题。网络图、热力图、雷达图——每一种都有成熟的理论框架可以引用（Tufte 的可视化设计原则、Bertin 的视觉变量理论），论文厚度直接翻倍。

**对产品**：一个嘻哈播放器如果连音频可视化都没有，就不配叫播放器。频谱动画和歌词光效是用户的第一印象。

## 三、Phase 1：歌词级可视化 + 推荐引擎基建（第 1–4 周）

### 3.1 目标

- 韵脚检测引擎完成，在歌词视图中用颜色标注韵脚
- 单首歌曲的「特征雷达图」可视化
- 推荐列表中每条推荐附带解释标签
- 说唱词库 v1 完成

### 3.2 可视化交付物

#### 3.2.1 韵脚颜色标注（LyricsView 增强）

**现状**：LyricsView.tsx 已支持 Apple Music 风格的 LRC 滚动同步。

**增强**：每行歌词末尾标注韵母，相同韵母的连续行使用同一颜色标记。韵脚颜色用渐变：连续押韵越长，颜色越饱和。

```
[紫色] 我来自街头 从不靠流量吃饭                (韵母: an)
[紫色] 他们靠人设 我靠实力翻盘                    (韵母: an)
[紫色] 你看不惯 但拿我没辙 只能干瞪眼看            (韵母: an)
[橙色] 换一个韵脚 换一种节奏                      (韵母: ou)
[橙色] 我的 Flow 像子弹 穿透了你的防守              (韵母: ou)
```

**实现**：在现有 LRC 行渲染逻辑中，对每行歌词结尾做 pinyin 查询，相邻押韵行编组，每组分配一个颜色。

**可视化价值**：这是整个系统最直接的用户感知入口。一个不听说唱的人看一眼也能理解「这首歌在押韵」。

> **论文素材**：「歌词韵脚可视化设计」——可以讨论颜色编码策略（为什么用色相区分韵脚组而不是用形状）、视觉信息密度的控制、以及如何适配不同类型歌词的显示密度。

#### 3.2.2 歌词特征雷达图

在歌词视图中新增一个「特征分析」按钮/分页，展示该歌曲的六维雷达图：

```
          韵脚密度
            /\
           /  \
    情绪强度   词汇丰富度
          |    |
    叙事性     俚语密度
           \  /
           \/
          节奏感
```

**实现**：纯 CSS/SVG 绘制雷达图，不需要引入 ECharts。六维特征的计算逻辑：

| 维度 | 计算方式 | 数据来源 |
|------|---------|---------|
| 韵脚密度 | 押韵行数 / 总行数 | Phase 1 韵脚检测 |
| 词汇丰富度 | 独特词数 / 总词数 | jieba 分词后计算 |
| 俚语密度 | 匹配说唱词库的词数 / 总词数 | 说唱词库匹配 |
| 叙事性 | 基于情感词典的叙事词评分 | 情感词典（正面/叙事类词汇） |
| 情绪强度 | 感叹句密度 + 攻击性词汇占比 | 规则匹配 |
| 节奏感 | 平均行长 + 行数密度 | 纯文本统计 |

> **论文素材**：「多维特征雷达图」的可视化设计——讨论为什么选雷达图而不是柱状图（因为维度之间需要对比，且雷达图天然适合展示「轮廓」和「指纹」），引用 Few 的多维数据可视化研究。

#### 3.2.3 推荐解释标签

在 DiscoverView.tsx 的推荐流中，每条歌曲下方增加一个半透明标签：

```
┌──────────────────────────────────────────┐
│ 🎵 《皇帝的新衣》 - AR 刘夫阳               │
│ ┌──────────────────────────────────────┐ │
│ │ 💡 你收藏的《消失的爱》韵脚密度接近        │ │
│ │    (0.78 vs 0.82)                     │ │
│ └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

标签类型与颜色映射：
- 韵脚接近 → 紫色标签
- 制作人相同 → 金色标签
- 厂牌关联 → 绿色标签
- 协作关系 → 蓝色标签
- 平台通用推荐 → 灰色标签

> **论文素材**：「推荐可解释性的视觉编码」——标签颜色作为视觉通道（Visual Channel），讨论颜色如何传达推荐置信度和推荐来源类型。

#### 3.2.4 技术选型

| 需求 | 工具 | 原因 |
|------|------|------|
| 韵脚检测 | js-pinyin | 纯前端，答辩不依赖后端 |
| 分词 | jieba-wasm 或内置词典 | 同上 |
| 雷达图 | 纯 SVG/CSS | 轻量，不需要第三方库，展示「手写」能力 |
| 韵脚标注 | CSS 类名 + 动态 style | 复用现有 LRC 渲染管线 |
| 推荐标签 | CSS 组件 | 复用现有组件系统 |

**为什么不用 ECharts/D3.js？**

Phase 1 只有一张雷达图，引入 ECharts（~1MB）杀鸡用牛刀。手工 SVG 雷达图既小又能体现工程能力——论文里可以有一段「可视化实现：手写 SVG 雷达图的数学推导」。
Phase 2 需要 ECharts 时再引入。

## 四、Phase 2：用户级可视化 + 智能洞察（第 5–8 周）

### 4.1 目标

#### 4.2.2 风格漂移桑基图

```
     ──────┐
            ├────────────── BoomBap (15% 迁移)

```

> **论文素材**：「桑基图在用户行为演变中的应用」——讨论流图（Flow Diagram）如何呈现用户偏好的时序演变。

#### 4.2.3 推荐路径可视化

在推荐流的每条歌曲下面，不只展示解释标签，还能展开看到完整的「推荐路径」：

```
你听过《皇帝的新衣》(AR) 
  → AR 与 Dough-Boy 合作过《Heat》 
    → Dough-Boy 制作了《摇滚主旋律》 
      → 推荐：Bakerie《香港地》
```

这个路径图用小型流程图画在侧边栏，每条弧线标注关系类型（「合作」「制作」「同厂牌」）。

> **论文素材**：「推荐路径的可视化表达」——知识图谱路径展示的经典问题，可以引用在电商推荐中的「路径解释」研究。

#### 4.2.4 此时引入可视化库

| 图表 | 库 | 原因 |
|------|------|------|
| 协作网络图 | ECharts（Force layout） | 内置力导向，支持交互 |
| 热力图 | ECharts（Heatmap） | 开箱即用 |
| 面积图/趋势图 | ECharts（Line + areaStyle） | 支持堆叠面积 |
| 桑基图 | ECharts（Sankey） | 原生支持 |

ECharts 在 Phase 2 引入是合理的——从一张图变成一套仪表盘，用框架统一管理比手写高效。论文里也可以对比「手写 SVG vs 可视化框架」的工程决策。

## 五、Phase 3：播放器动效 + 社区可视化（答辩后）

### 5.1 播放器动效

#### 5.1.1 Canvas 音频频谱（NowPlayingView 增强）

使用 Web Audio API 的 `AnalyserNode` 获取实时频域数据，在 NowPlayingView 背景绘制一个优雅的频谱动画。

**设计方向**（二选一）：
- **风格 A — Apple Music 风格**：低频到高频的水平频谱条，随 BPM 脉动，颜色从播放器主色渐变
- **风格 B — Spotify Canvas 风格**：粒子效果，低频膨胀高频旋散

推荐风格 A：暗色玻璃态 + 细频谱条，跟 Jasmine 现有设计语言融合。

#### 5.1.2 歌词光效

当前 LRC 滚动同步已实现，增加：
- 当前唱到的行：光晕扩散效果
- 韵脚切换时：微妙的颜色闪烁（押韵组之间过渡）
- Hook/副歌段：背景亮度微升

**技术路线**：CSS transition + `text-shadow` 动画，不需要 Canvas。

#### 5.1.3 Beat/BPM 同步

利用 Web Audio API 检测 BPM，让：
- 频谱动画的更新速率匹配 BPM
- 韵脚标注闪烁在强拍上触发
- 封面图随 Bass 做微缩放（bass-pumping effect）

**注意**：BPM 检测是一个有学术价值但工程上有难度的特征。可以做简化版（BPM 手动标注 + 客户端同步），论文里讨论 BPM 检测算法（自相关 vs 频域分析）但不一定完整实现。

### 5.2 社区可视化

标签系统、协作歌单、信誉网络的可视化——这些在答辩时只画概念图，不做实现。
