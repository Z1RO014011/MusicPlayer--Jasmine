## 一、韵脚颜色标注

### Q1：单押/双押/三押能否实现？

技术上可以。pinyin 库可以覆盖中文字符到韵母的转换。普通韵脚只要检测韵母是否相同即可。双押是把最后两个字分别取韵母，看是否都匹配。多音字影响在边缘情况下，说唱高频字可以建一个 20 行的校正表应对。

三押同理，复杂度随押韵位置数线性增长，不算瓶颈。真正的多押判断需要识别「多个韵脚位置是否同时押韵」，这个 pinyin 库能做，但算法逻辑比简单的韵母同一性检测多一步。

### Q2：英文说唱韵脚检测如何实现？

英文押韵和中文的本质区别：中文看韵母（拼音），英文看音位（不取决于拼写）。例如 "rough" 和 "stuff" 拼写不同但押韵，"bomb" 和 "tomb" 拼写相似但不押韵。

**解决方案**：使用 CMU Pronouncing Dictionary（卡内基梅隆发音词典），将 13 万个英文单词映射到 ARPAbet 音素。提取最后一个元音音素和后续辅音，相同即判断为押韵。这个词典是几百 KB 的文本文件，纯静态下载到前端直接使用，不需要后端服务。

**代价**：需要维护两套词库——中文用 pinyin，英文用 CMU 音素。代码量约多 60 行。但这是多语言支持的卖点，可写入论文。

### Q3：颜色标记是否会造成视觉混乱？

如果直接在文字上着色，十几行歌词七八种颜色确实会像彩虹。

**正确的设计方案**：颜色不放在文字上，而是放在每行的左侧竖线上（4px 宽）。相同韵脚组共用同一种左侧竖线颜色，文字保持白色（当前行）和半透明灰白色（非当前行）。颜色数量控制在 4–6 种，超过就循环复用。左侧竖线只在非当前行时显示，当前行不显示，保持焦点在正在唱的那一行。

这样既不干扰阅读，又能让用户扫一眼看出押韵结构。CSS 改动量约 15 行，不影响现有设计语言。

## 二、推荐解释标签

### Q4：网易云元数据是否能真正支撑推荐解释？

部分能用，但要清楚边界。

`/simi/song` 返回相似歌曲但不给推荐理由。能用来自动生成解释标签的元数据包括：是否同歌手、是否同专辑、歌手标签。这些能产生「同歌手」「同专辑《xxx》」等解释，但它们只是描述关联关系，不代表推荐质量的增量。

真正的高价值标签需要自己构造：厂牌关系（手工映射文件）、歌词相似度（TF-IDF）、用户行为信号（听歌历史分析）。这些是网易云没有且不会提供的数据。

### Q5：各实现方式的可行性与适合度

| 方式 | 可行性 | 适合度 | 说明 |
|------|--------|--------|------|
| 网易云元数据 | ✅ 高 | 中 | 80行代码，零成本，但标签浅层 |
| 手工映射文件 | ✅ 高 | 高 | 最符合当前开发阶段，120行，标签精准 |
| 用户行为信号 | ✅ 高 | 高 | IndexedDB数据现成，120行，有论文价值 |
| 爬虫→JSON | ✅ 中 | 中 | 边际效益存疑，花时间爬不如手工标注核心数据 |
| 歌词TF-IDF | ✅ 中 | 中高 | 需要jieba-wasm，150行，可支撑论文NLP章节 |
| 后端DB+API | ❌ 过早 | 低 | 过度工程化，Phase 3社区时再考虑 |

**建议**：先做手工映射文件 + 用户行为信号 + 网易云元数据的三源混合方案。三个来源合并到一条标签里，让推荐理由更丰满。

## 三、用户可视化

### Q6：可视化对用户到底重不重要？

分场景：

- **毕业答辩**：极其重要。图表是信息传递的最快方式，每张图能撑答辩PPT上的2分钟。
- **日常使用**：中等重要。大多数人打开播放器是为了听歌，但 Spotify Wrapped 证明了「有仪式感的数据可视化」有极强传播力。
- **关键原则**：不要在日常界面里堆图表，把可视化放在独立的统计面板或详情页里，用户想看才点进去。

### Q7：给曲库打标签的工作量

非常大。如果给每首歌打多维度标签（Flow风格、韵脚水平、内容类型、情绪等），几百首歌就是几百小时。

**实际做法**：不要给整个曲库打标签。先给种子数据集打——挑 50 首你收藏的歌，每首手工标 5–6 个标签。这 50 首作为论文「人工标注数据集」的实验基础，可以写标注一致性分析（Kappa系数）和标签分布统计。后续如果接入社区功能，用户标注会自然补充其余歌曲。

### Q8：推荐路径可视化的技术难点

核心难点是**数据不存在**。画推荐路径（如「你听了A → A和B合作 → B制作了C → 推荐C」）需要的协作数据、制作人数据，网易云API不直接提供。

如果用手工映射文件，路径就有了来源——你知道某个Rapper所属厂牌、厂牌下其他Rapper、合作过的歌手。

技术层面的难点：
- **力导向图布局稳定性**：ECharts内置解决，前端用 `useMemo` 缓存
- **节点缩放时性能**：限制渲染Top 50 Rapper
- **路径长度可读性**：限制3跳以内展示

## 四、新功能想法

### Q9：无缝切换

技术上完全可行。标准做法是用 Web Audio API 的 `AudioContext` + `AudioBufferSourceNode`，前一首末尾和后一首开头有几十毫秒的淡入淡出叠加。也可以用两个 `<audio>` 元素轮流播放。

**代价**：需要重构 PlayerContext 的播放管线，从 HTML5 Audio 切换到 Web Audio API，约 200–300 行改动。如果毕设主线是嘻哈推荐，这不是主线功能，可在「未来工作」中提及。

### Q10：和竞品的功能差异

**已有优势**：
- 暗黑OLED玻璃态UI（辨识度极高）
- Apple Music风格的LRC滚动同步
- 歌词编辑功能
- 键盘全快捷键
- 单曲分享卡片生成下载
- 中英文切换
- Electron桌面端三平台安装包
- 网易云VIP音质接入

**差异化重点**：聚焦「嘻哈垂直推荐」，而不是追Spotify的功能列表。

### Q11：歌手页可以加什么？

网易云 `/artist/detail` 返回 `briefDesc`、`/simi/artist` 返回相似歌手，结合手工映射文件中的厂牌归属，可以加入：
- 歌手简介
- 厂牌归属
- 相似歌手
- 热门歌曲Top 5

改动约 100 行代码。

### Q12：专辑页面可以加什么？

网易云 `/album` 返回 `picUrl`、`description`、`publishTime`、`company`、`artists`、`songs`。可以创建 `AlbumDetailView`：大图做背景，半透明叠加文字，下面是曲目表格。设计语言与 NowPlaying 保持一致。约 200 行代码。

### Q13：进一步接入网易云用户数据

NeteaseCloudMusicApi 已支持的接口：`/user/playlist`（所有歌单）、`/user/record`（听歌记录）、`/user/subcount`（收藏数）、`/album/sublist`（收藏专辑）、`/user/detail`（用户等级/关注/粉丝）、`/user/follows`（关注的歌手）、`/user/event`（动态）。

可实现在 Jasmine 中展示用户的完整网易云音乐档案——不是替代网易云，而是**网易云的升级版界面 + 嘻哈专用推荐层**。数据量可能很大，需要分页加载，首次登录先展示歌单和收藏，听歌记录按需加载。
