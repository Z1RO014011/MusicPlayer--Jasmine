# Jasmine 项目需求分析

## 文档信息

- 项目名称：Jasmine
- 文档主题：项目需求分析
- 文档类型：毕业设计风格正式分析文档
- 生成日期：2026-06-22

## 1. 项目概述

Jasmine 是一个基于 Vite、React、TypeScript 与 Electron 构建的桌面音乐播放器项目。该项目并不局限于基础音频播放，而是尝试围绕本地音乐管理、在线音乐接入、歌词交互、歌单组织、桌面端发布和产品化体验，构建一个具有完整使用闭环的音乐播放系统。

从当前项目文档、代码结构与开发记录可以看出，Jasmine 的目标并不是实现单一的“试听工具”，而是设计一个兼顾本地资源与在线内容、兼顾使用体验与工程可扩展性的综合型播放器。项目既服务于实际听歌场景，也承担毕业设计展示、论文撰写和桌面软件发布的复合目标。

因此，Jasmine 的需求分析应从“完整音乐消费流程”的角度展开，即围绕用户如何导入音乐、发现音乐、组织音乐、播放音乐、管理状态以及跨平台使用软件，系统性分析其功能需求与非功能需求。

## 2. 项目目标与建设意义

### 2.1 项目目标

Jasmine 的总体目标是构建一个具有较高完整度的桌面音乐播放器系统，使用户能够在统一界面中完成本地音乐管理、在线音乐搜索与发现、歌曲播放控制、歌词查看、歌单组织和历史记录追踪等操作，并在桌面端获得稳定、流畅、可持续使用的音乐体验。

### 2.2 建设意义

从产品角度看，Jasmine 试图将传统本地播放器与在线音乐平台的常见能力进行融合，使系统既具备本地资源管理能力，又具备网络化内容发现能力。

从工程角度看，项目涉及前端界面开发、状态管理、数据持久化、第三方 API 集成、桌面应用封装以及多音源抽象设计，具有较强的综合性。

从毕业设计角度看，该项目具备清晰的需求边界、明确的模块划分以及较强的展示效果，适合在论文中展开系统分析、架构设计、模块设计与测试验证等内容。

## 3. 目标用户分析

### 3.1 本地音乐用户

这类用户拥有本地音频文件，希望导入歌曲、长期保存、创建歌单并进行日常播放。对于此类用户，系统应重点满足本地音乐导入、元数据读取、音频持久化、歌曲检索和歌单管理等需求。

### 3.2 在线音乐用户

这类用户希望通过搜索、推荐、排行榜、艺人页、专辑页等方式浏览在线音乐内容，并能够直接播放相关歌曲。对于此类用户，系统应具备在线搜索、内容发现、扫码登录、推荐内容展示和在线播放等能力。

### 3.3 桌面应用用户

这类用户更关注软件是否具备独立安装、稳定运行、跨平台支持和良好交互体验。他们希望播放器像成熟桌面软件一样可下载安装、重复使用，并具有快捷键、播放状态记忆等桌面端典型特征。

### 3.4 毕业设计展示场景中的评审对象

除真实用户外，项目还面向毕业设计答辩场景中的教师和评审者。对于这一类“展示型观察者”，系统应具有完整的功能链路、较好的界面质感、明确的模块结构和可论证的技术路线，以支撑论文写作和现场演示。

## 4. 使用场景分析

### 4.1 本地音乐导入与管理场景

用户从本地选择多首音频文件导入系统，系统自动解析歌曲名、歌手、专辑和时长等元数据，并将音频文件保存到 IndexedDB 中，将歌曲信息和业务数据持久化到本地存储。用户随后可以在音乐库中查看全部歌曲、本地歌曲以及自建歌单，并进行删除、收藏和播放等操作。

### 4.2 在线音乐搜索与发现场景

用户进入发现页或搜索页，输入关键词后获取歌曲、歌单、艺人和专辑等搜索结果，也可以浏览热门搜索、推荐歌单和排行榜内容。若用户希望访问更完整的网易云内容，则可通过二维码完成登录并获取个性化推荐或用户歌单。

### 4.3 日常播放与沉浸式收听场景

用户点击任意歌曲后，系统进入播放状态，并提供暂停、上一首、下一首、进度拖拽、音量调节、随机播放、循环模式等控制能力。用户还可以进入全屏播放页查看歌词、编辑歌词、查看相似歌曲，并生成歌曲分享卡片。

### 4.4 歌单组织与收藏场景

用户可以创建歌单、重命名歌单、删除歌单、为歌单设置封面，并将歌曲添加到歌单中。系统还应支持“喜欢的音乐”这种特殊收藏机制，使用户能够以更低操作成本管理偏好歌曲和已收藏专辑。

### 4.5 桌面端持续使用场景

用户作为长期使用者，希望关闭应用后再次打开时仍然能够恢复之前的播放状态、音量、队列和语言偏好，同时希望软件具备安装包形式并可在不同操作系统下运行。这要求系统兼顾可安装性、可恢复性与平台兼容性。

## 5. 功能需求分析

### 5.1 音乐库管理需求

系统应支持用户导入本地音频文件，并自动读取歌曲元数据，包括歌曲名称、歌手、专辑、时长、封面颜色等信息。系统应提供音乐库界面，展示全部歌曲、本地歌曲、自建歌单、收藏专辑和播放历史，并允许用户对歌曲进行删除、播放和收藏等操作。

### 5.2 歌单管理需求

系统应支持用户创建歌单、删除歌单、修改歌单名称，并允许从音乐库向歌单中添加歌曲或从歌单中移除歌曲。为了提升个性化体验，系统还应允许用户通过上传图片或选择渐变色的方式设置歌单封面。

### 5.3 在线音乐接入需求

系统应接入在线音乐服务，至少支持歌曲、歌单、艺人和专辑的搜索，并能够获取在线歌曲的播放地址和歌词。系统还应提供推荐歌单、排行榜、热门搜索和分类歌单等发现型入口，以满足用户从“搜索已知内容”到“浏览未知内容”的完整在线发现需求。

### 5.4 用户登录需求

系统应支持通过二维码扫码方式登录在线音乐平台。登录后，系统应能够检测和恢复登录状态，持久化用户 Cookie，并在此基础上获取更完整的用户相关内容，例如个性化推荐、用户歌单以及更高质量的音频播放能力。

### 5.5 播放控制需求

系统应提供完整的播放控制功能，包括播放、暂停、上一首、下一首、进度跳转、音量调节、随机播放和循环模式切换。系统还应维护统一播放队列，以保证不同页面进入播放时能形成一致的播放上下文。

### 5.6 播放队列需求

系统应支持展示当前播放队列，并允许用户进行添加到队列、下一首播放、移除队列歌曲和清空队列等操作。对于用户而言，播放队列不仅是当前播放上下文的容器，也是串联本地音乐、在线歌单和专辑播放的重要机制。

### 5.7 歌词与沉浸式播放需求

系统应支持 LRC 歌词解析与同步显示，并在全屏播放页中以滚动高亮形式呈现歌词。若歌曲没有歌词，系统应给出明确提示；若用户拥有可编辑歌词文本，则应允许手动录入和保存歌词。系统还应在播放页中展示歌曲详细信息、相似歌曲入口以及分享能力，以增强沉浸式体验。

### 5.8 收藏与偏好管理需求

系统应允许用户快速标记“喜欢”的歌曲，并通过特殊歌单形式集中管理偏好内容。同时，系统应支持在线专辑的收藏与取消收藏，并能够在音乐库中集中展示已收藏专辑。

### 5.9 播放历史需求

系统应自动记录用户的播放历史，并按时间维度进行分类展示，例如今天、昨天、本周和更早。该功能可帮助用户回看近期听过的内容，也为后续推荐、统计和用户画像模块提供数据基础。

### 5.10 国际化与设置需求

系统应支持中英文界面切换，并将用户语言偏好持久化保存。设置模块当前至少需要承担语言切换职责，未来还可扩展到主题、缓存、播放策略和下载策略等更多配置项。

### 5.11 分享与传播需求

系统应支持将当前播放歌曲生成可视化分享卡片，并导出为 PNG 文件。该功能可增强项目的产品感和展示效果，也有助于提升系统的完整度和差异化。

### 5.12 桌面发布需求

系统应支持基于 Electron 的桌面封装与构建发布，形成可安装的桌面应用。系统应能够面向 macOS 和 Windows 输出安装包，并具备桌面环境下的 API 调用能力和资源访问能力。

## 6. 非功能需求分析

### 6.1 可用性需求

系统应具备清晰的导航结构和统一的交互逻辑，使用户能够在发现、音乐库、搜索、设置和播放页之间自然切换。界面风格应统一、信息层级应清晰，常用操作应具备低学习成本。

### 6.2 性能需求

系统在本地歌曲加载、歌单切换、在线播放和歌词展示时应保持较好的响应速度。对于在线大歌单、分页拉取、音频 URL 获取等操作，系统应具备渐进加载、批量请求和预取优化能力，以降低用户等待成本。

### 6.3 数据持久化需求

系统应保证本地文件数据、歌曲元数据、歌单信息、歌词、历史记录、播放状态和登录状态可以在应用重启后恢复，以支持长期使用场景。持久化方案应兼顾文件体积、访问性能与实现复杂度。

### 6.4 可扩展性需求

系统的在线音源层应具备抽象能力，使后续接入其他音乐平台时无需大规模重写现有业务逻辑。当前项目已经建立音源接口和注册层，说明系统在设计上具备一定扩展导向。

### 6.5 可维护性需求

系统应采用模块化结构，将界面组件、状态管理、持久化逻辑、API 封装和工具函数进行合理拆分，以降低后续迭代和排错成本。对于毕业设计项目而言，可维护性还直接影响论文中系统架构设计部分的表达质量。

### 6.6 兼容性与发布需求

系统应兼容主流桌面操作系统，至少包括 macOS 和 Windows。构建流程应稳定，发布产物应便于下载安装，开发环境与桌面运行环境之间的差异应有明确处理方案。

## 7. 系统模块划分

结合现有实现情况，Jasmine 可划分为以下核心模块：

### 7.1 界面与导航模块

该模块负责整体布局、页面切换、侧边栏导航、移动端布局适配以及不同视图之间的跳转控制，是系统用户体验的外层骨架。

### 7.2 播放器核心模块

该模块负责播放状态管理、播放控制、队列调度、随机与循环模式切换，以及当前歌曲与播放进度同步，是整个系统的核心业务模块。

### 7.3 本地音乐管理模块

该模块负责本地文件导入、元数据提取、音频文件存储、歌曲信息持久化和本地音乐展示，是支撑本地听歌场景的关键模块。

### 7.4 在线音乐服务模块

该模块负责封装网易云相关接口，包括搜索、歌单详情、歌曲地址、歌词、推荐、排行榜、艺人和专辑等访问能力，是在线内容接入的核心模块。

### 7.5 歌单与收藏模块

该模块负责自建歌单、喜欢的音乐、收藏专辑等内容的组织与管理，是用户个性化内容沉淀的主要承载者。

### 7.6 歌词与沉浸展示模块

该模块负责歌词解析、同步滚动、全屏播放页展示、歌词编辑以及相似歌曲联动，是提升系统体验和展示效果的重要模块。

### 7.7 持久化存储模块

该模块负责 IndexedDB 与 localStorage 的协同使用，实现文件级与元数据级数据的长期保存，是应用可恢复性的基础。

### 7.8 桌面封装与发布模块

该模块负责 Electron 主进程与渲染进程之间的通信、桌面资源访问、打包构建和安装包发布，是项目从 Web 应用升级为桌面软件的关键支撑。

## 8. 当前实现情况分析

从现有代码和开发日志来看，Jasmine 已经实现了较高完成度的基础需求，主要体现在以下几个方面：

第一，系统已经完成本地音乐导入、播放控制、歌单管理、歌词展示、收藏、历史记录、国际化和桌面封装等主要功能，能够形成较完整的用户使用闭环。

第二，系统已具备较强的在线内容接入能力，包括网易云搜索、推荐、排行榜、扫码登录、用户歌单、艺人页和专辑页等，这意味着项目已经从“本地播放器”扩展为“在线内容播放器”。

第三，系统已预留多音源抽象层，并引入 YouTube 相关依赖，说明项目在架构上已经考虑后续扩展，而非一次性编码实现。

总体而言，Jasmine 当前已经不再是功能验证阶段，而是进入“功能稳定化 + 体验增强 + 垂直深化”的阶段。

## 9. 当前需求缺口与问题分析

尽管项目完成度较高，但从需求落地角度看，仍存在若干关键缺口。

### 9.1 大型在线歌单播放稳定性问题

当前最明显的问题是网易云大歌单，特别是 2000 首以上歌单，在点击播放时仍可能出现进度条停留在 0、无声音等异常。这说明在线播放链路在大数据量上下文中仍存在稳定性不足的问题。对于一个以在线播放为重要卖点的播放器而言，该问题属于高优先级缺陷。

### 9.2 多音源能力尚未完全落地

尽管系统已具备音源抽象接口，但目前实际稳定接入并在界面中被完整使用的仍主要是网易云音源。因此，多音源支持目前更多体现为架构准备，而不是用户可直接感知的成熟功能。

### 9.3 设置模块能力偏弱

当前设置页主要聚焦语言切换，尚未形成完整的系统设置中心。若从成熟桌面播放器角度衡量，未来仍需补充主题切换、缓存清理、下载目录、默认音质、快捷键说明等设置项。

### 9.4 队列管理仍有增强空间

现有队列已具备查看和基础管理能力，但在拖拽排序、批量删除、保存为歌单、来源标识等方面仍有较大提升空间。该模块也是后续增强产品专业度的重要方向。

### 9.5 个性化推荐与数据分析能力仍处于早期阶段

项目已经开始向“嘻哈垂直推荐”和“用户行为可视化”方向演进，并补充了相似歌曲、用户画像相关 API，但这些能力尚未形成成熟的用户可见模块。若作为毕业设计亮点继续深化，仍需进一步落地推荐解释、统计视图和可视化分析页面。

## 10. 后续优化建议

结合当前项目状态，后续优化可按照“先稳定、再增强、后亮点”的顺序推进。

### 10.1 优先级一：保证核心播放链路稳定

首先应重点修复大歌单播放异常、在线音频地址获取与播放上下文同步等问题，保证在线播放功能在主要使用场景中足够稳定。这是系统可信度与可用性的基础。

### 10.2 优先级二：完善桌面播放器常用能力

在核心稳定后，应继续增强播放队列、设置中心、播放状态记忆、最近播放、迷你播放器等典型桌面播放器能力，使软件更贴近日常使用习惯。

### 10.3 优先级三：扩展系统差异化亮点

在基础体验稳定后，可继续推进多音源接入、歌词效果增强、推荐解释、用户统计面板和可视化分析模块。这些内容不仅能提升产品辨识度，也能为论文和答辩提供更强的展示亮点。

## 11. 结论

综合分析可知，Jasmine 是一个目标清晰、模块完整、实现深度较高的桌面音乐播放器项目。其需求核心并非单纯实现音频播放，而是围绕“本地音乐管理、在线音乐发现、沉浸式播放体验和桌面端长期使用”建立完整系统。

从当前实现情况看，项目已经完成了主要功能需求的落地，具备较强的展示性与工程综合性；从后续演进角度看，系统仍存在在线播放稳定性、多音源落地、设置体系完善和个性化能力深化等提升空间。

如果将该项目用于毕业设计论文撰写，则可将其描述为一个兼顾实用性、扩展性与展示效果的综合型桌面音乐播放器系统，并在需求分析、系统设计、模块实现和测试优化等章节中继续展开论述。
